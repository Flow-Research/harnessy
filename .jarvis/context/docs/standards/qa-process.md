# QA Process

## Purpose

This document is the canonical Harnessy-level QA process contract.

It defines the shared rules for:

- regression specs as source of truth
- canonical scenario IDs and fields
- semantic feature catalogs and result snapshots
- test discovery and header requirements
- drift validation
- browser walkthrough and DB assertion expectations
- security scenarios as first-class regression coverage
- code generation expectations
- test-quality validation expectations

Project repos can extend this contract with repo-specific profiles, seed flows,
environment rules, and result sinks, but they should not redefine the shared
baseline semantics documented here.

## Core Model

Harnessy QA has three layers:

1. **Regression specs** describe what should be tested.
2. **Tests** implement those scenarios in code.
3. **Runtime validation** detects drift and coverage gaps.

The canonical deterministic runtime command is `qa`. The older `flow-qa`
command remains a backward-compatible alias for existing projects.

Mature repositories may add a fourth operational surface:

4. **Result sinks** mirror execution status into committed snapshots,
   spreadsheets, dashboards, or release scorecards.

Result sinks are project extensions. They must not become the source of truth
for scenario intent.

## Source Of Truth

- Regression specs are the source of truth for scenario intent.
- Tests must map back to canonical scenario IDs.
- Drift between specs and tests is a defect, not a documentation nicety.

Generated or derived artifacts should have one owning command and should not be
hand-edited when the runtime owns them.

## QA Profile Contract

Harnessy resolves QA configuration from a repo-local JSON profile. Default lookup order:

1. `.harnessy/qa-profile.json`
2. `.flow/qa-profile.json`
3. `qa/qa-profile.json`

The profile must define:

- regression spec paths
- app IDs
- browser test roots
- API test roots
- optional output paths such as coverage reports

The profile may also define:

- feature catalog paths
- run-result snapshot directories
- browser walkthrough artifact directories
- security findings directories
- result sinks such as spreadsheets or dashboards
- local, staging, and production execution commands

The QA profile is the canonical runtime input. Adapter metadata in files such as
`.flow/delivery-profile.json` may enrich codegen or validation, but does not
replace the QA profile as the source of truth.

## Feature Catalog Contract

Larger repos should maintain a semantic feature catalog that maps scenario ID
prefixes to product features. The recommended artifacts are:

- `qa/features.generated.yaml` — deterministic generated catalog
- `qa/features.overrides.yaml` — hand-maintained semantic overrides
- `qa/features.changelog.md` — append-only rationale for catalog changes

Each feature entry should include:

- stable slug
- human-readable name
- canonical `id_prefix`
- owning apps
- source regression suites or spec files
- result snapshot path, when snapshots are used
- optional external result-sink identifier

`/qa-feature-catalog` owns semantic review of this layer. Generated files should
be regenerated by repo-local tooling; agents should not hand-edit generated
catalog files.

Prefix stability matters. Renaming an `id_prefix` with existing committed
results or external rows requires an explicit migration plan.

## Canonical Scenario IDs

Scenario IDs must use the canonical form:

```text
<PREFIX>-<NNN>
```

Rules:

- `PREFIX` is 2-5 uppercase alphanumeric characters starting with a letter
- `NNN` is a zero-padded three-digit sequence number

Examples:

- `AUTH-001`
- `MATCH-014`
- `E2E-005`

Non-canonical IDs are drift defects.

## Minimum Spec Fields

Harnessy regression specs should use `##` headings plus a top field block.

Minimum expected fields:

- `Layer:`
- `Status:`
- `Test File:` when implemented or scaffolded

Common optional fields:

- `Linked Refs:`
- `User Flow:`
- `Role:`
- `Route:`
- `Mode:`
- `Type:`
- `DB Assert:`
- `Expected:`
- `Security Class:`
- `Threat Actor:`
- `Attack Surface:`

## Layers

Shared layer vocabulary:

- `browser`
- `api`
- `security`
- `manual`

Meaning:

- `browser`: user-visible workflows and browser automation
- `api`: backend logic, authorization, workflow transitions, DB invariants
- `security`: adversarial security scenarios
- `manual`: human-only scenarios with no automated implementation

Security scenarios are first-class regression rows, not sidecar prose. They
must include `Security Class:` and should be generated or reviewed through
`/qa-security-sweep` whenever a feature introduces new attack surface.

Allowed security classes:

- `authn`
- `authz`
- `injection`
- `xss`
- `csrf`
- `crypto`
- `ratelimit`
- `businesslogic`
- `exposure`
- `supplychain`
- `config`
- `other`

## Browser Walkthrough Contract

Browser tests should be grounded in observed application behavior. When
selectors, redirects, auth state, responsive behavior, or UI state are
uncertain, run a Playwright walkthrough before writing or updating tests.

Walkthrough artifacts should record:

- role or actor
- route attempted
- actual URL after redirects
- visible headings, forms, buttons, links, tables, and empty states
- selector strategy and confidence
- screenshots or traces for ambiguous/failing states

Codegen may use source code for orientation, but it should not silently guess
browser selectors or navigation behavior. Unknown selectors become TODOs or
walkthrough blockers.

## DB Assertion Contract

Browser scenarios that verify persisted mutations, workflow transitions, access
control, or data visibility should include `DB Assert:` with the post-condition
the automated test should check.

Examples:

- row count changes
- workflow state changes
- ownership/tenant isolation
- generated artifact existence
- absence of leaked rows for unauthorized roles

If direct DB validation is unsafe or unavailable for an environment, the spec or
result notes should say so explicitly. Skipping DB checks without explanation is
a false-green risk.

## Run-Result Snapshot Contract

Repos may commit run-result snapshots such as `qa/run-results/<feature>.json`.
Snapshots are the committed mirror of execution status and manual notes; they
are not the scenario source of truth.

Snapshot rows should be keyed by canonical scenario ID and may include:

- automation status
- pass/fail/skip/not-run
- severity
- notes or actual result
- last run timestamp
- environment-specific results

External result sinks, for example spreadsheets, should be synchronized from
specs and snapshots by repo-local commands. A lower number of updated tabs or
rows does not itself mean tests cannot pass together; it usually means the sync
scope, selected features, missing result rows, or missing external tabs limited
the writeback.

## Test Header Contract

Automated test files must include top-level annotations near the start of the file:

```ts
// @qa-spec: qa/<layer>/scripts/<file>.md
// @qa-suite: <PREFIX> (<slug>)
```

Missing `@qa-spec` or `@qa-suite` headers are quality defects.

## Drift Contract

`qa drift` is the shared deterministic drift gate.

At minimum, it must surface:

- spec parse errors
- implemented scenarios without tests
- tests referencing nonexistent spec scenarios
- files missing `@qa-spec` / `@qa-suite` headers

Teams may add stricter project-local rules, but these are the baseline Harnessy-wide defects.

## Codegen Contract

`/browser-integration-codegen` and `/api-integration-codegen` must:

- treat the QA profile as the canonical regression input
- preserve canonical scenario IDs
- use adapter metadata only for imports, fixtures, helper modules, and project-specific test shims
- leave explicit TODOs instead of guessing selectors, assertions, or auth setup
- use Playwright walkthrough artifacts for browser selectors when available
- require or preserve `DB Assert:` coverage for persistence-sensitive browser flows

Codegen should not silently invent non-canonical scenarios or bypass runtime drift expectations.

## QA Sweep Contract

`/qa-sweep` is the shared full-cycle QA orchestrator. It should:

- run deterministic preflight with `qa ids`, `qa tests`, and
  `qa drift`
- discover source surfaces and compare them with regression coverage
- use `/browser-qa` or repo-local Playwright helpers for browser walks
- map positive, negative, unauthorized, boundary, destructive, and persistence scenarios
- coordinate `/qa-feature-catalog` when new prefixes or semantic merges appear
- coordinate `/qa-security-sweep` for new attack surface
- coordinate codegen skills and `/test-quality-validator`
- report skipped, blocked, and not-run scenarios with reasons

The sweep is allowed to use repo-local execution and result-sink commands, but
those commands remain project extensions.

## Validator Contract

`/test-quality-validator` must treat these as quality defects:

- missing `@qa-spec` / `@qa-suite` headers
- non-canonical scenario IDs
- `Status: implemented` scenarios without matching tests
- tests referencing nonexistent specs
- browser tests for persistent flows without DB assertions or an explicit
  environment limitation

It should combine those runtime defects with:

- coverage completeness
- false-green risk detection
- role and authorization coverage
- project adapter rule violations when configured

## Recommended Workflow

1. Write or update regression specs.
2. Run `qa ids` to confirm the spec parses.
3. Generate or update tests.
4. Run `qa tests` and `qa drift`.
5. Run `/test-quality-validator` for higher-level quality review.
6. Fix drift or validator findings before claiming QA completeness.

For broad coverage expansion, use:

1. `/qa-sweep` to discover, walk, map, generate, execute, and report.
2. `/qa-feature-catalog` when feature names, prefixes, or result sinks drift.
3. `/qa-security-sweep` when a feature, diff, or PR adds attack surface.

## Project Extensions

Project repos can extend the shared contract with:

- seed and login preparation flows
- remote environment policies
- result sinks such as spreadsheets or dashboards
- app-specific role inventories
- local runbooks for release QA
- feature-scoped execution commands such as `plan`, `execute`, and `sync`
- environment policies for destructive tests, DB assertions, and production smoke

Those extensions should live in the project repo, not in this shared standard.
